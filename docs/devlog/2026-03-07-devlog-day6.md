## Device Registration System — Design & Build

### Motivation

Before shipping a consumer OS, there needs to be an optional facility for users to register an install, for us to be able to ascertain how many devices are running MahalaOS. Without data, it's impossible to make the case to funders (NLnet), partners (repair shops), or the community that MahalaOS has real-world traction.

### Privacy-first design

The registration system was designed with the same philosophy as the OS itself — collect the absolute minimum, make it genuinely anonymous, and be transparent about what is stored.

**What is stored:**
- `device_id` — SHA-256 hash of hardware identifiers with a `mahalaos:` salt. One-way: cannot be reversed to recover MAC address or serial number.
- `first_seen` — UTC timestamp of first registration
- `last_seen` — UTC timestamp of most recent registration (updated on reflash)
- `reg_count` — number of times this device has registered

**What is explicitly not stored:**
- IP addresses
- Device model or manufacturer
- Any personally identifiable information

**UUID derivation priority:**
1. MAC address (`wlan0` / `eth0` / `usb0`)
2. CPU serial
3. `/etc/machine-id` fallback
4. Random UUID (last resort)

Reflashing the same device updates `last_seen` rather than creating a duplicate row — upsert on `device_id`.

### Files produced

| File | Purpose |
|---|---|
| `server/device_uuid.py` | Generates and caches hardware-derived UUID at `/var/lib/mahalaos/device-uuid` |
| `server/registration_client.py` | Wizard-side POST to `register.mahalaos.org/v1/device`. Fire-and-forget, 10s timeout. Writes result to `/var/lib/mahalaos/registration-status` |
| `server/registration_server.py` | Flask API — POST `/v1/device`, GET `/v1/stats`, GET `/health` |
| `server/requirements.txt` | `flask`, `flask-limiter`, `gunicorn` |
| `render.yaml` | Render deployment config (repo root, `rootDir: server`) |

### API endpoints

```
POST /v1/device          — anonymous device registration
GET  /v1/stats           — returns {"registered_devices": N}
GET  /health             — health check
```

Rate limiting: 5 registrations per IP per hour via `flask-limiter`.

## Deployment — Render + Cloudflare

### Infrastructure decisions

- **Hosting:** Render basic tier web service 
- **Database:** SQLite on a 1GB Render persistent disk at `/var/lib/mahalaos` (~$0.25/month)
- **DNS:** Cloudflare (nameservers transferred from 123reg)

### Deployment process

The deployment surfaced several issues worth documenting:

**Issue 1 — Docker detection**
Render attempted to use Docker on first deploy, ignoring `render.yaml`. Fixed by manually setting runtime to Python and specifying root directory and start command in the Render dashboard.

**Issue 2 — Disk not attached**
First redeploy after adding the disk returned `unable to open database file`. Disk was visible in Render settings but not yet mounted. Fixed by triggering a fresh deploy after confirming disk attachment.

**Issue 3 — `init_db()` never called by gunicorn**
The table creation function was only called inside `if __name__ == "__main__"` — gunicorn imports the module directly and never hits that block. The `devices` table therefore never existed, causing every request to return `{"error": "Server error"}`. Fixed by moving `init_db()` to module level so it runs on import regardless of how the server is started.

**Issue 4 — Nameserver propagation**
The `register` CNAME was added to Cloudflare but Render could not verify the domain. Root cause: 123reg was still authoritative — Cloudflare DNS records were being ignored entirely. Fixed by transferring nameservers to Cloudflare at 123reg. Propagation took ~15 minutes.

### Final state

```
https://register.mahalaos.org/v1/stats  →  {"registered_devices": 0}
https://register.mahalaos.org/v1/device →  accepts POST with UUID
https://register.mahalaos.org/health    →  {"status": "ok"}
```

## Landing Page — mahalaos.org

### Design brief

- **Goal:** Equal weight for developers/contributors and end users
- **Sections:** Features, Install, Contribute, Newsletter
- **Tone:** Clean and minimal — let libadwaita do the visual work ethos applied to web
- **Tagline:** *"Your phone, your rules."*

### Implementation

Single-file `index.html` with no external dependencies beyond Google Fonts. Logo embedded as base64 data URI — no broken image on first load, no additional HTTP request.

**Colour palette** extracted directly from the logo:
- Primary green: `#158644`
- Dark green: `#106e36`
- Light green: `#27a55c`
- Gold accent: `#cbc067`

**Sections built:**
- **Hero** — tagline, description, animated phone mockup showing the setup wizard screen, two CTAs (Get Started / GitHub)
- **Why MahalaOS** — 6-card grid: Privacy by default, Consumer-ready UX, Free as in free, Repairability first, Calls & SMS that work, Upstream first
- **How to install** — 4-step guide with terminal code blocks, sticky device compatibility note
- **Contribute** — 3-card dark section: code, issues, distribution partnership
- **Newsletter** — email signup with client-side confirmation
- **Footer** — logo, copyright, links to GitHub and postmarketOS

**Animations:** CSS scroll-triggered fade-up on all major elements. Phone mockup floats on a 6-second sine curve.

### Hosting

Deployed via Cloudflare Pages direct upload (bypassing GitHub OAuth loop issue with the MahalaOS organisation). Custom domain `mahalaos.org` configured in Cloudflare Pages settings.

**GitHub OAuth loop** — documented for future: Cloudflare Pages struggled to authorise against a GitHub organisation repo. Workaround was direct asset upload. Auto-deploy via Git can be reconnected later.
