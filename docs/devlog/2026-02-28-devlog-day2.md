## User Experience Observations

These are raw friction points observed during normal use — the kind of things a non-technical user would notice immediately.

- **Flashing process is too hard for consumers.** Multiple challenges including EDL mode and a crash on first boot. An online guide and full browser interface would be needed for non-technical users. This is a known gap and a Phase 2 priority.
- **Front camera is broken.** Too zoomed in and rendering in black and white.
- **Back camera is partially working.** Image is fuzzy with a green tint. Shutter speed seems acceptable. Gallery gesture navigation needs improvement.
- **Screen goes black during calls.** No UI visible except volume controls. No way to access the dialler, speaker toggle, or end call button without the power button.
- **Echo/feedback on calls.** A test contact reported hearing their own voice back. Likely a PipeWire mic routing issue — microphone not muting properly when the other party speaks.
- **Inconsistent swipe-to-go-back.** Some screens support the swipe gesture, others don't. This is a known upstream issue with GNOME Control Centre ([#3608](https://gitlab.gnome.org/GNOME/gnome-control-center/-/issues/3608)) — different panels were built at different times with different GTK navigation widgets. Flagged upstream, no fix scheduled yet.
- **No haptic feedback when typing.** Expected on a touchscreen phone. Needs investigating.
- **No tap-to-wake.** Screen only wakes on the power/lock button. Common expectation for smartphone users.
- **Boot unlock uses full keyboard instead of numpad.** When entering the LUKS disk encryption passphrase at boot, users are presented with a full QWERTY keyboard rather than a numeric keypad. After boot, the GNOME lock screen correctly shows a numpad. The inconsistency is confusing.

  **Investigation:** After extracting and inspecting the initramfs, the boot unlock screen is handled by `buffyboard` (part of the `buffybox` package), spawned from `init_functions.sh`. buffyboard has no numeric/PIN mode — only full QWERTY. An upstream feature request has been filed: [buffybox issue](https://gitlab.postmarketos.org/postmarketOS/buffybox/-/issues).

## Technical Investigation: Waydroid Networking

**Time spent:** ~2 hours. Should be zero for users.

### Problem

Waydroid container running but showing `IP: UNKNOWN`. No internet connectivity. WhatsApp and all Android apps non-functional.

### Root Cause

postmarketOS's nftables firewall (`inet filter` table) has `policy drop` on both `input` and `forward` chains. No rules exist for the `waydroid0` bridge interface. This blocks DHCP traffic between the Android container and the host's dnsmasq — the container never receives an IP address, default route, or DNS server.

### Symptoms

- `waydroid status` shows `IP: UNKNOWN`
- Manually assigning an IP inside Android allows ping to raw IPs, but apps still fail
- Android uses policy routing (`ip rule`) with a catch-all `unreachable` rule at priority 32000 — routes must be in Android's named routing tables, not the main table
- DNS is empty: `DnsAddresses: [ ]`

### The Fix

Two nftables rules on the host:

```bash
sudo nft add rule inet filter input iifname "waydroid0" accept
sudo nft add rule inet filter forward iifname "waydroid0" accept
```

Once applied, dnsmasq automatically assigns IP, gateway, and DNS to the container. No manual configuration inside Android required.

### Persistence

A systemd service (`mahala-waydroid-net.service`) has been created to apply these rules at boot, before the Waydroid container starts.

**Note:** On postmarketOS, `nft` lives at `/usr/sbin/nft`, not `/usr/bin/nft`. The service was initially broken due to the wrong path — corrected before committing.

This fix has been committed to the MahalaOS repo and is MahalaOS's first real upstream-relevant discovery.
